fix(anti-fraud): recompute ARV against the CORRECTED address in one pass

Bug: Gemini computed the ARV during the main qualification pass, which ran with
the WRONG (agent-submitted) address's comps. The address auto-correction
happened afterward and re-pulled Zillow data, but the ARV still reflected the
wrong property until a second manual re-run.

Now, when the AI flags an address discrepancy, the correction branch re-pulls
Zillow for the address heard on the call AND immediately re-runs a focused ARV
report (new runArvReport — a small text-only Gemini call over the corrected
comparables) so the ARV range, narrative, and comps reflect the RIGHT property.
The verdict already used the corrected market value; now everything — corrected
address, corrected Zillow data/comparables, and ARV — is consistent in a single
analysis pass. No discrepancy → no extra call (unchanged path).

Refactored the market-data prompt formatter into a shared marketDataText()
helper used by both the qualification prompt and the ARV recompute.

Verified: next build green, tsc --noEmit exit 0.
