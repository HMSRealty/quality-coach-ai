feat(inbound): Inbound API & webhook ingestion for external dialers (Step 6)

POST /api/inbound/lead (edge): accepts { address, seller_name, campaign_id,
audio_url } with a Bearer API key. Authenticates by sha-256 of the key against
public.api_keys, downloads the audio_url into the private call-recordings bucket,
registers call_uploads, creates the lead, and fires the SAME /api/leads/analyze
pipeline as the manual form. Smart duplicate bypass: same-address lead that was
previously Disqualified/Error is revived & re-analyzed; any other status -> 409.
Audio is pulled before the lead is created so failed downloads never orphan a row.

New migration 0011_api_keys.sql: per-user api_keys table (sha-256 hash + prefix,
label, last_used_at, revoked) with owner-only RLS; the inbound route reads it
via the service role.

Integrations UI /dashboard/settings/api (Clean Enterprise): generate API keys
(raw key shown once, only the hash is stored), revoke/delete, the inbound
webhook URL, and copyable JSON + cURL payload examples. Sidebar "Webhooks &
Integrations" now routes here.

ACTION REQUIRED: run supabase/migrations/0011_api_keys.sql.
